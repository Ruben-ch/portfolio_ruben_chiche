Serveur Web Nginx

Reverse Proxy & Hébergement Sécurisé

Debian 11 Nginx SSL/TLS Reverse Proxy

Contexte du Projet

Dans le cadre de l'architecture réseau sécurisée (VLAN 50 - Zone Serveurs), le serveur Web est un composant critique. J'ai choisi la solution Nginx plutôt qu'Apache pour sa légèreté et sa capacité à gérer un grand nombre de connexions simultanées grâce à son architecture asynchrone.

Ce serveur a une double fonction : il héberge les applications internes (Site Vitrine, Intranet) et agit comme un Reverse Proxy pour sécuriser et fluidifier l'accès aux services backend comme l'outil de ticketing GLPI.

Rôles & Fonctionnalités

Reverse Proxy

Le serveur intercepte les requêtes des clients et les redirige vers les services appropriés. Cela masque l'architecture interne et ajoute une couche de sécurité.

Virtual Hosts

Utilisation des "Server Blocks" pour héberger plusieurs sites (ex: glpi.entreprise.local et www.entreprise.local) sur une seule adresse IP.

Sécurité HTTPS

Mise en place de certificats SSL/TLS pour chiffrer les communications. Redirection automatique du trafic HTTP (port 80) vers HTTPS (port 443).

Configuration Technique

Voici un extrait commenté de la configuration d'un Virtual Host agissant en tant que Reverse Proxy pour le service GLPI. Notez la gestion des en-têtes (headers) pour transmettre les bonnes informations au backend. 

server {
    listen 80;
    server_name glpi.entreprise.local;
    return 301 https://$host$request_uri; # Redirection HTTPS forcée
}

server {
    listen 443 ssl;
    server_name glpi.entreprise.local;

    ssl_certificate /etc/nginx/ssl/glpi.crt;
    ssl_certificate_key /etc/nginx/ssl/glpi.key;

    location / {
        proxy_pass http://192.168.50.30:80; # IP du serveur GLPI interne
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
    }
}

Bilan Personnel

La mise en place de ce serveur Nginx m'a permis de comprendre la différence fondamentale entre un proxy et un reverse proxy. J'ai également pu appréhender la gestion des certificats SSL (générés via OpenSSL pour le labo) et l'importance des fichiers de logs (/var/log/nginx/access.log) pour le débogage lors des erreurs 502 Bad Gateway.
